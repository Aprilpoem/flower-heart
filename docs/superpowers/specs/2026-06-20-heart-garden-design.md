# 心事花园设计文档

## 目标

心事花园是一个本地私密网页应用，用来把个人经历中的事件、瞬间和感受写下来，并转化成一朵朵花。

它不是效率工具，也不是情绪打卡工具。它更像一个小型恢复空间：用户写下一个瞬间，就像种下一朵花；随着记录变多，花园逐渐长满，成为一种可见的证据：我经历过、命名过、保存过，也仍然在生长。

## 产品定位

心事花园默认私密，允许选择性分享。

用户平时只给自己看完整花园；当某一朵花足够想被别人看见时，可以把这一朵导出成分享卡片。产品的重点不是发布、社交或获得反馈，而是让用户拥有一种可控的暴露方式：递出一朵花，而不是开放整座花园。

## 第一版范围

第一版只支持四件核心事情：

1. 写下一个瞬间，形成一条标本记录。
2. 保存后，这条记录在花园中变成一朵花。
3. 在标本册中回看、整理这些记录。
4. 将单独一朵花导出成可分享卡片。

第一版不做账号、登录、云同步、公开主页、评论、点赞、社交动态、提醒、连续打卡、复杂统计或 AI 分析。

## 核心隐喻

- 事件/瞬间：一颗种子。
- 写下来：种下一朵花。
- 保存的记录：花园里的花。
- 回看记录：在花园中散步。
- 导出分享：递出一朵花，而不是开放整座花园。

产品文案可以使用“种一朵花”“标本册”“花园”等词，但整体气质要克制、温柔、成人化，避免过度可爱或幼稚。

## 主要视图

### 花园视图

花园视图是首页，也是应用的第一屏。用户打开应用后直接看到自己的花园，而不是营销页或说明页。

每一条记录对应一朵花。花朵的颜色来自用户选择的心情。第一版可以使用 CSS 绘制的抽象花朵或图标式花朵，不追求真实植物插画，但要让用户感到“这是我种下的东西”。

花园视图需要支持：

- 点击或轻触一朵花，打开这朵花的标本详情。
- 通过主操作入口“种一朵花”创建新记录。
- 通过页签或导航切换到标本册视图。

花园的重点是“积累感”。即使只有几朵花，也要显得安静、完整；花多起来之后，要有一种个人景观逐渐生成的感觉。

### 标本册视图

标本册是更适合阅读和整理的视图。它以卡片或列表形式展示每一朵花的完整记录。

每张标本卡包含：

- 标题。
- 日期。
- 事件/瞬间。
- 当时的感受。
- 它让我想到什么。
- 标签。
- 心情颜色。

第一版支持按时间浏览，并支持按标签筛选。搜索功能暂不作为第一版必须功能。

## 记录模型

每一朵花包含以下字段：

- `id`：稳定唯一标识。
- `title`：这朵花的名字。
- `createdAt`：创建时间。
- `moment`：发生了什么，或我注意到了什么。
- `feeling`：我当时感到了什么。
- `association`：这件事让我想到什么。
- `tags`：短标签，例如工作、健康、关系、身体、游泳、虚无、希望。
- `mood`：一个心情值。

第一版心情选项：

- 焦虑。
- 悲伤。
- 平静。
- 快乐。
- 愤怒。
- 麻木。
- 希望。

每种心情对应一种花朵颜色。色彩需要有区分度，避免整个应用只像单一色系的情绪墙。

## 种花流程

用户点击“种一朵花”后进入轻量编辑界面。

编辑界面包含：

- 标题。
- 事件/瞬间。
- 感受。
- 联想。
- 标签。
- 心情颜色。

保存后，应用回到花园视图，并立即出现新花。这种即时反馈是第一版的核心成就感来源。

第一版不强制做草稿自动保存。但如果用户在未保存且已有内容时关闭编辑界面，需要提示确认，避免误丢内容。

## 分享卡片

每一朵花都可以单独导出为分享卡片。

分享卡片包含：

- 花朵视觉。
- 标题。
- 日期。
- 心情颜色。
- 正文节选或完整事件文本。
- 可选标签。

分享必须由用户主动触发。应用不会自动发布任何内容，也不会开放整个花园。

第一版优先做“可复制文字卡片”和“卡片预览”。如果技术成本可控，再增加 PNG 图片下载；如果 PNG 导出复杂，则放到后续版本。

## 存储

第一版使用浏览器本地存储保存数据，优先选择 `localStorage`，因为它足够简单、无需服务器、适合快速完成可用版本。

应用需要在一个不打扰的位置说明：数据只保存在当前设备和浏览器中，除非用户主动导出，否则不会上传。

第一版不需要后端服务。

## 视觉方向

应用应该安静、亲密、成熟。

设计原则：

- 第一屏直接呈现花园。
- 文字阅读要舒服，留白充足。
- 不做幼稚游戏感。
- 不做重型效率仪表盘。
- 不做连续打卡、积分、排行榜、成就徽章。
- 常见操作优先使用清晰图标和简短文案。
- 卡片和按钮尺寸稳定，移动端和桌面端都不能出现文字重叠。

花园要有表达力，但不能喧闹。它应该像一个能让人停下来呼吸的地方。

## 错误处理

应用需要处理以下情况：

- 必填内容为空时，用温和方式提示缺失项。
- 本地保存失败时，展示清晰提示，并保留用户已经输入的内容。
- 分享导出失败时，提供可复制文字卡片作为兜底。
- 本地数据异常时，尽量保留可读取记录，并提示有部分数据无法加载。

## 验证方式

手动验证需要覆盖：

- 创建一朵花。
- 新花出现在花园中。
- 点击花朵能打开详情。
- 同一条记录能在标本册中看到。
- 标签筛选可用。
- 分享卡片可以预览或复制。
- 刷新页面后记录仍然存在。
- 桌面端和移动端布局正常。

如果使用 JavaScript 框架，自动化测试优先覆盖记录模型、本地存储工具、创建和读取流程。

## 第一版实现建议

推荐第一版做成轻量本地网页应用。

技术选择建议：

- 使用静态 HTML/CSS/JavaScript 或轻量 React 应用。
- 使用 `localStorage` 保存记录。
- 使用 CSS 或内联 SVG 绘制抽象花朵。
- 分享功能先做卡片预览和复制文本，PNG 下载作为可选增强。

第一版的成功标准不是功能完整，而是用户愿意打开、愿意写、写完真的觉得自己种下了一朵花。
